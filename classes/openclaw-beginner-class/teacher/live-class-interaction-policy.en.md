# Live Class Interaction Policy (Teacher/TA)

## Purpose
Keep class focused, reduce interruptions, and ensure timely student support.

## Roles
- Instructor: leads class flow and final decisions
- Sarah (TA lead): support orchestration and triage
- Aether (TA assist): executes within Sarah-approved framework
- Students: follow channel/mode rules for questions and support

## Interaction Modes

### 1) Lecture Mode
- Goal: uninterrupted concept teaching/demo
- Allowed: logistics clarifications, whole-class urgent blockers
- Not allowed: deep individual debugging in main thread

### 2) Practice Mode
- Goal: hands-on execution support
- Allowed: hints, scaffolds, troubleshooting
- Required support ladder: Hint → Scaffold → Walkthrough

### 3) Q&A Mode
- Goal: open discussion, clarification, and extension
- Allowed: all class-relevant questions

## Channel Rules
- Google Meet: live instruction and verbal Q&A
- Telegram: quick support, status checks, blockers
- GitHub: source of truth (materials, homework, issue logs)

## Stuck Protocol (Student must provide)
1. Goal
2. What they tried
3. Actual output/error
4. Expected output
5. Best guess

## Triage Priority
1. Whole-class blockers
2. Repeated issues across students
3. Individual blockers
4. Nice-to-have optimizations

## Response Targets
- Urgent live blocker: <2 minutes
- Practice-mode first response: <5 minutes
- Deferred issue via GitHub: <24 hours

## Boundaries
- TA does not override instructor flow in Lecture Mode
- Aether execution starts after Sarah framework approval
- Avoid answer dumping when process learning is the objective

## Logging and Follow-up
After each class, log:
- top 3 blockers
- fixes applied
- material updates needed

## Enforcement Line
"Switching to Lecture Mode — hold technical debug for Practice Mode."
